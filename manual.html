pinchfin User Manual

CRE Underwriting Intelligence Platform

v1.0 — February 2026 — github.com/pinchfin

1. What pinchfin Is

pinchfin is an open source commercial real estate underwriting intelligence system. It is a collection of skills, deterministic calculation engines, a regulatory RAG corpus, and workflow templates designed to support institutional-quality credit analysis.

It was built by a lender for actual lending operations. Over two years of development alongside real underwriting workflows, running against 60+ detailed loan analyses, 38+ live loans, and ongoing analysis campaigns across 1,700+ pipeline loans. It was not built in a lab. It was built on a couch, mostly at night, and refined every time something broke on a real deal.

At its core, pinchfin provides:

  • 200+ individual skills for CRE underwriting tasks, each with its own documentation, expected inputs, outputs, and examples
  • 23+ modules that chain related skills into structured analysis sequences
  • Deterministic Python calculation engines for loan sizing, DSCR, debt yield, LTV, cash flow analysis, and waterfall constraints
  • 5,900+ RAG documents sourced from OCC, FDIC, FNMA, Freddie Mac, HUD, CREFC, SIFMA, and other regulatory and trade bodies
  • API integrations with Census Bureau, BLS, FRED, FEMA, Walk Score, and Google Maps
  • A three-agent quality system (Producer, Checker, Resolver) modeled after institutional credit committee processes
  • Phased context management designed around the practical constraints of large language model context windows

pinchfin is designed to run inside Claude Code, Anthropic's agentic coding environment. It is not a standalone application. It is not a SaaS product. It is a toolkit — a structured, documented, battle-tested collection of components that you can use whole, in parts, or as a reference for building your own systems.

What this means in practice You clone a repository. You configure some API keys. You open a terminal. You tell the system what you want to analyze. It runs through a structured sequence of research, calculation, and synthesis steps, and it produces institutional-quality output. The math is deterministic. The analysis is grounded in regulatory source material. The output is formatted for committee consumption.

2. What pinchfin Is Not

Use at Your Own Risk

pinchfin is released in the spirit of collaboration. It is not a regulated financial tool and does not constitute investment, legal, tax, or accounting advice. The following statements are non-negotiable terms of use.

  • Not investment advice. Nothing produced by this system should be construed as a recommendation to make or refrain from making any investment decision.
  • Not a replacement for human judgment. Commercial real estate underwriting is contextual and judgment-driven. This system assists the underwriting process. It does not replace it. A human-in-the-loop is required at every stage.
  • Not production-hardened enterprise software. There is no enterprise security stack, no SOC 2 compliance, no dedicated infrastructure team, and no SLA.
  • Not a regulated financial tool. pinchfin has no regulatory certification, no compliance attestation, and no fiduciary status.
  • Outputs must always be independently verified. The AI model may hallucinate. API data may be stale or inaccurate. Calculations depend on input quality. Every output should be validated against independent sources before being relied upon for any business decision.
  • API dependencies may fail or return inaccurate results. Third-party data sources change their schemas, rate-limit requests, and occasionally go offline. This is normal. Plan for it.

4. Getting Started

This section walks you through the complete setup process from a fresh machine to a working pinchfin installation. Each step is sequential. Do not skip ahead.

Step 1: Install Prerequisites

Before cloning pinchfin, confirm that Node.js, Python, and Git are installed on your system.

# Check Node.js (need 18+)
$ node --version

# Check Python (need 3.10+)
$ python --version

# Check Git
$ git --version

If any of these are missing, install them from their respective official sources: nodejs.org, python.org, git-scm.com.

Step 2: Install Claude Code

Claude Code is Anthropic's terminal-based AI development environment. It is the runtime that executes pinchfin's skills and workflows.

# Install Claude Code globally via npm
$ npm install -g @anthropic-ai/claude-code

After installation, verify it works by running claude in your terminal. You will need an Anthropic API key configured — Claude Code will prompt you for this on first launch.

Step 3: Clone the Repository

# Clone the pinchfin repository
$ git clone https://github.com/pinchfin/pinchfin.git
$ cd pinchfin

Step 4: Install Python Dependencies

# Install required Python packages
$ pip install -r requirements.txt

This installs the packages required by the deterministic calculation engines and data service integrations.

Step 5: Configure API Keys

# Copy the example environment file
$ cp .env.example .env

# Open .env in your preferred editor and add your API keys
# Each key is labeled with the service it belongs to

The .env file contains clearly labeled placeholders for each API key. Fill in the ones you have. Leave the others blank — the system will skip integrations for missing keys and tell you what data it could not retrieve.

Security Never commit your .env file to version control. The .gitignore is pre-configured to exclude it, but verify this before pushing any changes. API keys are credentials. Treat them accordingly.

Step 6: Verify Installation

# Run the verification script to confirm everything is configured
$ python engines/verify_setup.py

This script checks that Python dependencies are installed, API keys are present, and the calculation engines can execute.

Step 7: Launch and Load

# Start Claude Code
$ claude

# Once inside Claude Code, load pinchfin
> /load pinchfin

The /load pinchfin command initializes the system, loads the credit framework, makes the skills and calculation engines available, and prepares the context management system. You are now ready to run analyses.

First run suggestion Start with something small. Try running a single skill like the DSCR calculator or the FRED rate puller to confirm your setup works end to end. Do not attempt a full underwriting workflow on your first session.

6. How to Run a Workflow

pinchfin workflows follow a phased approach. Each phase has entry requirements, a defined sequence of skills, and exit criteria that must be met before advancing. The system cannot skip phases. This is intentional — upstream analysis quality directly determines downstream output quality.

1
Load Deal Context

Upload deal documents (rent roll, T12 operating statement, organizational documents, appraisal, prior analyses). The system reads and indexes these files, extracts key data points, and establishes the factual foundation for everything that follows. The credit framework is loaded automatically.

2
Run Core Analysis Skills

Execute the relevant module sequence. For a standard multifamily deal, this typically means: Sponsor Analysis, Market Analysis, Comparable Analysis, Core Underwriting, Cash Flow, and Loan Sizing. Each module runs its constituent skills in order. Each skill produces a structured artifact that feeds the next.

3
Risk Assessment and Synthesis

The Risk Analysis module aggregates all upstream outputs, runs stress scenarios, identifies concentration issues, flags deal-specific risk factors, and produces a synthesized risk assessment with a recommendation.

4
Document Assembly

The Document Assembly module compiles all analysis into formatted committee-ready output: the IC memo, term sheet, and supporting exhibits. The Quality Assurance system reviews the final package before delivery.

Example: Running a Market Analysis

Once pinchfin is loaded, you interact with it through natural language in Claude Code.

# After loading pinchfin, request a market analysis
> Run a market analysis for the Phoenix MSA. Pull Census demographics,
  BLS employment data, and FRED economic indicators. Include the
  multifamily supply pipeline and vacancy trends.

# The system will:
# 1. Call Census API for demographic data (population, income, housing)
# 2. Call BLS API for employment and labor force data
# 3. Call FRED API for economic indicators (rates, CPI, permits)
# 4. Synthesize into a structured market profile
# 5. Flag any concerning trends or data gaps
# 6. Output a formatted Market Analysis artifact

Example: Running the Loan Sizing Engine

# Run the waterfall constraint engine with specific parameters
> Size this loan using the following assumptions:
  - NOI: $1,250,000
  - Cap rate: 5.75%
  - Interest rate: 6.50%
  - Amortization: 30 years
  - Max LTV: 75%
  - Min DSCR: 1.25x
  - Min debt yield: 8.0%

# The Python engine runs all five constraints:
# DSCR constraint, LTV constraint, debt yield constraint,
# max loan amount, IO coverage
# The binding constraint determines max proceeds
What to expect Analyses take time. A full market analysis with multiple API calls might take 30 to 90 seconds. A complete deal underwriting across all phases can take several minutes. This is normal. The system is making real API calls, running real calculations, and producing structured output — not generating plausible-sounding text.

7. Understanding the Output

Reading a Generated IC Memo

The investment committee memo follows a standardized structure designed for institutional consumption. Each section traces to upstream analysis artifacts.

  • Executive Summary — One-page deal overview with the recommendation, key metrics (DSCR, LTV, debt yield), and critical risk factors
  • Sponsor Overview — Entity structure, financial strength assessment, and track record analysis with source citations
  • Market Analysis — MSA and submarket profile with demographic, employment, and supply/demand data sourced from government APIs
  • Property Analysis — Physical description, condition assessment, comparable rent and sale analysis
  • Financial Analysis — T12 normalization, cash flow projections, sensitivity tables
  • Loan Structure — Sizing waterfall results, proposed terms, constraint analysis
  • Risk Assessment — Identified risks with mitigants, stress test results, concentration analysis

Interpreting Risk Flags

The system generates risk flags at multiple severity levels.

LevelMeaningRequired Action
CRITICALDeal-breaker condition per the credit frameworkStop. Review the condition manually. The system will not advance past this flag without explicit override.
WARNINGOutside standard parameters but not disqualifyingReview and document. The analysis continues but the flag appears in the final output for committee attention.
INFONotable data point or deviation from expectationsAwareness only. Included in supporting documentation for context.

Validating Calculation Results

Every calculation produced by the Python engines includes a full audit trail. The output shows each input variable, the formula applied, intermediate steps, and the final result. To validate manually:

  1. Check the input values against your source documents
  2. Run the same calculation in a spreadsheet or financial calculator
  3. Compare the intermediate steps, not just the final answer
  4. Verify that the binding constraint in the loan sizing waterfall matches your independent analysis

Where Logs Are Stored

All analysis sessions produce logs in the logs/ directory within the pinchfin project folder. Each session creates a timestamped log file containing API calls made, data received, calculations performed, and any errors encountered. These logs are essential for audit trails and debugging.

Comparing Versions

When you re-run an analysis with updated assumptions or new data, the system generates a new artifact alongside the previous version. You can compare versions by reviewing the artifacts in the output/ directory, where each is timestamped and labeled with the deal identifier and analysis phase.

8. The Calculation Layer

Why Math Is in Python, Not AI

Large language models approximate. They round things they should not round. They skip steps in calculations and fill in the gap with something plausible. In lending, plausible is not good enough.

The entire math layer runs in deterministic Python, completely separated from the AI reasoning layer. The calculation engines are hardcoded, tested, and produce the same output for the same input every time. The AI handles research, synthesis, and narrative. Python handles numbers. There is no overlap.

The Credit Box Configuration

All calculation parameters are defined in engines/credit_box_config.json. This is the single source of truth for your lending parameters.

// Example structure of credit_box_config.json
{
  "dscr_minimum": 1.25,
  "ltv_maximum": 0.75,
  "debt_yield_minimum": 0.08,
  "io_dscr_minimum": 1.50,
  "max_loan_amount": 25000000,
  "amortization_years": 30,
  "interest_rate_floor": 0.05,
  "stress_rate_buffer": 0.02,
  "vacancy_assumption": 0.05,
  "management_fee_pct": 0.04,
  "replacement_reserves_per_unit": 300
}

To modify your credit parameters, edit this file directly. Changes take effect on the next calculation run. No code changes are required.

Running Standalone Calculations

The calculation engines can be run independently of the full pinchfin system. This is useful for quick sizing checks or spreadsheet validation.

# Run the loan sizing engine directly
$ python engines/loan_sizer.py --noi 1250000 --rate 0.065 --cap 0.0575

# Run the cash flow engine
$ python engines/cash_flow_engine.py --input data/sample_t12.json

Validating Against Manual Calculations

The engines output full intermediate steps. For a DSCR calculation, you will see the annual debt service computation (principal + interest on the amortization schedule), the NOI used, and the resulting ratio. Compare each step against your own calculator or spreadsheet, not just the final number.

9. The RAG Corpus

What Is Included

The RAG (Retrieval-Augmented Generation) corpus contains 5,900+ documents that ground every analysis in actual regulatory and market frameworks. Without this corpus, the AI model would generate analysis based on its training data — which may be outdated, incomplete, or wrong. The corpus ensures that when the system references lending guidelines, it is citing real documents, not hallucinating policies.

Sources include:

CategorySourcesCoverage
Banking RegulatorsOCC, Federal Reserve, FDICCRE lending guidance, examination handbooks, risk management bulletins, supervisory letters
GSEsFNMA, Freddie MacMultifamily lending guides, eligibility requirements, underwriting standards, seller/servicer guides
GovernmentHUD, FHAMAP Guide, affordable housing requirements, fair market rents, income limits
Trade AssociationsCREFC, SIFMA, MBA, IREM, CCIMReporting standards, market conventions, best practices, industry definitions

How It Grounds Analysis

When the system analyzes a deal, it retrieves relevant sections from the corpus based on the property type, loan program, jurisdiction, and risk factors involved. For example, when underwriting an FNMA multifamily loan, the system pulls the specific eligibility and underwriting requirements from the current DUS Guide rather than relying on the model's general knowledge of agency lending.

How to Add Your Own Documents

The corpus is extensible. To add your own documents:

  1. Place document files (PDF, TXT, DOCX) in the appropriate subdirectory under rag-corpus/
  2. Follow the existing naming conventions: source-organization_document-title_version.ext
  3. Run the indexing script to process and chunk the new documents: python rag-corpus/index.py
  4. New documents become available to the system on the next session load

Chunking and Indexing

Documents are split into semantically meaningful chunks designed to fit within the retrieval context window while maintaining coherence. The chunking strategy varies by document type: regulatory documents are chunked by section and subsection; guidelines are chunked by topic; reference materials are chunked by definition or entry. Each chunk retains metadata about its source, section, and publication date for citation purposes.

11. Troubleshooting

This section covers the most common issues and their resolutions.

API Key Errors

Symptom: Authentication failures, 401/403 responses, or "API key not found" messages in logs.

Fix: Open your .env file and verify the key for the failing service. Keys expire, get rotated, or are sometimes entered with trailing whitespace. Copy the key fresh from the provider's dashboard. Run python engines/verify_setup.py to test all keys at once.

Context Window Exceeded

Symptom: The system stops mid-analysis with a context length error, or outputs become truncated.

Fix: Break your analysis into smaller phases. Load fewer documents at a time. Run individual modules rather than the full workflow in a single session.

Calculation Mismatches

Symptom: The DSCR, LTV, or loan amount from the Python engine does not match your spreadsheet.

Fix: Check credit_box_config.json to confirm the parameters match your assumptions exactly. Common mismatches include: amortization period (years vs. months), interest rate (annual vs. monthly), and NOI definition (before or after reserves). The engine output shows all intermediate steps — compare each one to find where the divergence occurs.

Missing Dependencies

Symptom: ModuleNotFoundError or ImportError when running Python scripts.

Fix: Run pip install -r requirements.txt again. If you are using a virtual environment, make sure it is activated.

Rate Limiting

Symptom: 429 responses from APIs, or data fields returning as null/empty when they should have values.

Fix: Wait 60 seconds and retry. Census and BLS allow around 500 requests per day on free keys. Google Maps has per-second and per-day limits.

Stale or Missing Data

Symptom: Demographic data looks outdated, or certain fields return as unavailable.

Fix: Government APIs (Census, BLS) update on their own schedule, often with a lag of several months to a year. The system uses the most recent available data. Check the source API documentation for coverage details.

Model Hallucinations

Symptom: The AI states a specific number, statistic, or fact that does not match any source data or API output.

Fix: This is expected behavior with language models. The RAG corpus and calculation engines minimize this, but it cannot be eliminated entirely. Every quantitative output should be validated against the calculation engine logs or API response data. If the AI generates a number that did not come from the Python engine or an API call, do not trust it.

12. Known Limitations

Honesty about what this system cannot do is as important as explaining what it can. These limitations are documented not as excuses but as facts that should inform how you use and rely on pinchfin.

  • Single-user system. pinchfin has no multi-user support, no user authentication, no role-based access control, and no concurrent session management. It is designed for one person running one analysis at a time.
  • No enterprise security stack. There is no encryption at rest beyond what your operating system provides, no audit logging beyond session logs, no intrusion detection, and no vulnerability scanning infrastructure.
  • No SOC 2 compliance. This system has not undergone any third-party security audit or compliance certification.
  • Context window constraints. Very large deal packages (hundreds of pages of documents) cannot be processed in a single pass. The phased context management system mitigates this but does not eliminate the limitation.
  • API rate limits. Free-tier API keys have request limits. Paid API tiers offer higher limits.
  • Not optimized for high concurrency. Running multiple instances simultaneously may cause API rate-limit conflicts, file-locking issues, or context corruption.
  • The model may hallucinate. Despite the RAG corpus, calculation engines, and quality gates, the underlying language model can and will occasionally generate information that is not grounded in source data. Always verify.
  • No real-time data. API data has inherent lag. Census data can be 1 to 2 years old. BLS data updates monthly. FRED data varies by series.
  • Limited property type coverage. The system was primarily developed for multifamily lending. Coverage for office, retail, industrial, and hospitality varies by module.
