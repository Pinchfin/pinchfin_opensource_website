Retrieval Methodology

Documents are retrieved using semantic similarity with metadata filtering. Each retrieval is logged with source attribution, relevance score, and citation context.

Semantic Similarity + Metadata Filtering

Queries are matched against the corpus using embedding-based semantic search, then filtered by document type, publication date, geographic relevance, and source tier. Results are ranked by composite relevance score.

Source Authority Hierarchy

Tier 1 (Primary): Institutional research from nationally recognized brokerage firms, licensed commercial data services, public records via county assessor and recorder offices, and federal statistical agencies. Primary sources require institutional-grade methodology, auditable data collection, and industry-standard accreditation. Tier 2 (Cross-reference): Established rental listing platforms and residential data aggregators. Validation and triangulation only — never sole source for institutional credit decisions. Tier 3 (Never cite): Content marketing, search-optimized editorial, crowd-sourced reviews, and unverified user-generated data. Excluded from all analytical outputs.

Data Freshness Management

Market data enforces a 60-day shelf life. Stale data triggers automatic re-retrieval before downstream consumption. Economic indicators (SOFR, Treasuries, unemployment) are pulled live via MCP data services at deal initiation.

Citation Requirements

Every analytical claim must trace to a specific source with publication date and access method. Unsourced assertions are flagged by the Checker agent. Citation format follows institutional research standards — no footnotes without provenance.

Comparable Sales Upload Workflow

Operators upload comparable sale and lease data for integration with the underwriting rent determination and valuation modules.

Upload Format

Structured CSV or JSON with required fields: property address, sale/lease date, price or rent per unit/SF, unit count, property type, and data source. Optional fields include cap rate, occupancy at sale, and buyer/seller identity.

Processing & Normalization

Uploaded comps are geocoded, validated against county assessor records where available, and normalized to per-unit and per-SF metrics. Duplicate detection flags properties already in the comp database.

Distance-Weighted Matching

Comps are scored by proximity to the subject property, vintage (time since transaction), size similarity, and property class alignment. Closer, more recent, and more similar comps receive higher weighting in the analysis.

Underwriting Integration

Validated comps flow into the rent determination module (Phase 6A) and the comparable analysis report. Vintage adjustments are applied automatically based on market trend data. All comp selections are documented with rationale.

Red Flag Detection (L1-L4)

LEVEL 1

Data Anomalies

Missing fields, format inconsistencies, calculation mismatches between documents, and data freshness violations.

LEVEL 2

Analytical Flags

Metrics outside expected ranges, trend reversals, comparable mismatches, and assumption conflicts across phases.

LEVEL 3

Structural Concerns

Sponsor concentration risk, market deterioration signals, regulatory exposure, and environmental contamination indicators.

LEVEL 4

Critical Alerts

Deal-breaking findings: fraud indicators, material misrepresentation, constraint waterfall failures, and policy violations.

Learning & Improvement

The QA improvement loop captures findings from every deal. The Producer agent reads the last 5 improvement log entries before each new deal, ensuring error patterns are not repeated. Version control tracks every corpus update, retrieval methodology change, and detection threshold adjustment.

Quality Assurance

Three-Agent QA system. Every output verified.

The Triple-Agent QA system uses three independent roles — Producer, Checker, and Resolver — to ensure analytical accuracy, arithmetic correctness, and internal consistency.

AGENT 1

Producer

Executes the analytical workflow. Generates all outputs, calculations, and narrative content. Reads the last 5 QA improvement log entries before starting each deal to avoid known error patterns.

AGENT 2

Checker

Independent review of all Producer outputs. Verifies arithmetic, cross-references data sources, checks constraint waterfall logic, validates assumption consistency, and flags discrepancies.

AGENT 3

Resolver

Arbitrates disputes between Producer and Checker. Makes final determinations on flagged items, documents resolutions, and updates the improvement log with learnings.

Deployment Intensity

TRIPLE

All three agents active. Full Producer → Checker → Resolver cycle. Used for IC-ready deliverables, final credit decisions, and high-stakes outputs.

RECOMMENDED

DOUBLE

Producer + Checker only. No Resolver arbitration. Used for intermediate outputs, draft analysis, and pipeline screening where speed matters.

SINGLE

Producer only. No QA review. Used only for exploratory analysis, rapid triage, and non-deliverable work product.

Quality Gates

Every phase output passes through four validation gates before downstream consumption.

Arithmetic Correctness

Independent recalculation of all numerical outputs from raw inputs. Debt service, coverage ratios, yield metrics, and amortization schedules are verified against deterministic computation. Tolerance: zero — any discrepancy triggers rejection.

Internal Consistency

Cross-reference between document sections: NOI in the cash flow must match NOI in the sizing waterfall. Rent roll totals must reconcile with revenue projections. Unit counts, square footage, and occupancy figures must be consistent across all deliverables.

Source Attribution

Every factual claim and market assertion must trace to a specific data source with publication date. Unsourced statements are flagged. Tier 3 sources trigger automatic rejection. Source freshness is validated against the 60-day shelf life policy.

Constraint Compliance

All sizing outputs are validated against the credit box configuration. LTV, DSCR, debt yield, LTC, and maximum proceeds are checked against their respective thresholds. Any constraint violation blocks the output from advancing to the next phase.

Deployment

Get running in five steps.

Technical Requirements

  • Claude Code CLI (Anthropic) with active API key
  • Node.js 18+ for MCP data services
  • Python 3.10+ for report generation engine
  • API credentials for integrated data services (geographic, demographic, economic, property-level intelligence)
  • Windows, macOS, or Linux

Setup Steps

STEP 1

Connect MCP Server

Register the pinchfin MCP server in your Claude Code configuration. The server provisions all skills, analytical engines, data services, and configuration files through a managed integration.

STEP 2

Configure data service credentials

pinchfin integrates with licensed data services for geographic, demographic, economic, and property-level intelligence. The platform manages all API connectivity, rate limiting, and data normalization. Credentials are stored locally and never transmitted beyond configured endpoints.

STEP 3

Initialize your credit box

Edit credit_box_config.json to reflect your firm's constraint thresholds, sizing parameters, and credit philosophy. The credit box is yours — it encodes your institutional standards. Institutional data — comparable sales packages, market studies, rent surveys, and third-party reports — is uploaded to the deal folder for each transaction. The platform processes, validates, and integrates your proprietary market intelligence alongside its research capabilities.

STEP 4

Verify MCP connectivity

Run the system validation check to confirm MCP server connectivity, data service availability, and configuration integrity. All 11 data tools across 10 services should report healthy.

STEP 5

Run your first deal

Start with the Context Brief (Phase -1) and progress through the execution framework. The system guides you through each phase with structured handoffs and blocking gate enforcement.

Credit Box Configuration

The credit box is the central configuration file that governs all constraint thresholds and sizing parameters. Here is a representative example:

credit_box_config.json JSON 
{
  "version": "3.0",
  "constraints": {
    "ltv_as_is":        { "max": 0.75, "type": "hard_ceiling" },
    "ltv_stabilized":   { "max": 0.75, "type": "hard_ceiling" },
    "dscr_stabilized":  { "min": 1.05, "type": "hard_floor"   },
    "dy_stabilized":    { "min": 0.07, "type": "hard_floor"   },
    "ltc":              { "max": 0.80, "type": "hard_ceiling" },
    "max_loan_amount":  100000000
  },
  "sizing": {
    "method": "constraint_waterfall",
    "binding_constraint": "minimum_of_all"
  }
}

Environment Considerations

pinchfin operates through a managed MCP integration. Analytical computation executes within your Claude Code environment. Deal data remains under your control — no proprietary information is stored on pinchfin servers beyond the MCP session context.

For production deployment at scale, consider: API rate limiting configuration for data services, concurrent session management across analysts, storage allocation for deal records and generated documents, and backup procedures for the master repository and credit box configuration.
