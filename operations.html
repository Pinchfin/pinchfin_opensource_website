Document Processing

Intake, parse, and normalize deal documents.

Offering Memo Parser

Extracts structured data from PDF/Word offering memoranda: property details, financial summaries, sponsor information, and market context.

Rent Roll Normalizer

Ingests rent rolls in any format, normalizes unit types, identifies concessions, flags anomalies, and produces standardized unit-level data.

T12 Expense Processor

Parses trailing twelve-month operating statements, categorizes line items, calculates per-unit metrics, and flags material variances.

Third-Party Report Indexer

Indexes appraisals, environmental reports, PCAs, and surveys into the deal record for cross-reference during analysis.

Industry benchmarks in OCR and data extraction are improving rapidly. Consult your IT department for enterprise options. The intake process includes a QA process with multiple agents and multiple reviews before finalizing the deal book.

Analytical Engine

Deterministic computation. No generative drift.

Constraint Waterfall

Five-constraint sizing engine: LTV As-Is, LTV Stabilized, DSCR, Debt Yield, and LTC. Runs all constraints and identifies the binding factor automatically.

Cash Flow Projector

Multi-year pro forma generation with dual-scenario modeling (base case and downside), revenue growth assumptions, and expense escalation.

Sensitivity Engine

Rate sensitivity, cap rate sensitivity, NOI stress testing, and scenario matrix generation with visual output tables.

Return Calculator

Levered and unlevered IRR, equity multiple, cash-on-cash return, and sponsor waterfall distribution analysis.

Reserve Sizer

Tax and insurance escrow calculation, replacement reserve sizing, and interest reserve determination based on business plan timeline.

Lender Sizer

Auto-generated lender sizing with live rate pulls from FRED, spread application, constraint waterfall execution, and term sheet structuring. Produces loan pricing summaries with binding constraint identification.

You must still check, verify, and validate all math, calculations, and formulae.

Retrieval Methodology

Documents are retrieved using semantic similarity with metadata filtering. Each retrieval is logged with source attribution, relevance score, and citation context. The system prioritizes recency (60-day shelf life for market data) and source authority.

Red Flag Detection (L1-L4)

LEVEL 1

Data Anomalies

Missing fields, format inconsistencies, calculation mismatches between documents, and data freshness violations.

LEVEL 2

Analytical Flags

Metrics outside expected ranges, trend reversals, comparable mismatches, and assumption conflicts across phases.

LEVEL 3

Structural Concerns

Sponsor concentration risk, market deterioration signals, regulatory exposure, and environmental contamination indicators.

LEVEL 4

Critical Alerts

Deal-breaking findings: fraud indicators, material misrepresentation, constraint waterfall failures, and policy violations.

Learning & Improvement

The QA improvement loop captures findings from every deal. The Producer agent reads the last 5 improvement log entries before each new deal, ensuring error patterns are not repeated. Version control tracks every corpus update, retrieval methodology change, and detection threshold adjustment.

Quality Assurance

Three-Agent QA system. Every output verified.

The Triple-Agent QA system uses three independent roles — Producer, Checker, and Resolver — to ensure analytical accuracy, arithmetic correctness, and internal consistency.

AGENT 1

Producer

Executes the analytical workflow. Generates all outputs, calculations, and narrative content. Reads the last 5 QA improvement log entries before starting each deal to avoid known error patterns.

AGENT 2

Checker

Independent review of all Producer outputs. Verifies arithmetic, cross-references data sources, checks constraint waterfall logic, validates assumption consistency, and flags discrepancies.

AGENT 3

Resolver

Arbitrates disputes between Producer and Checker. Makes final determinations on flagged items, documents resolutions, and updates the improvement log with learnings.

Deployment Intensity

TRIPLE

All three agents active. Full Producer → Checker → Resolver cycle. Used for IC-ready deliverables, final credit decisions, and high-stakes outputs.

RECOMMENDED

DOUBLE

Producer + Checker only. No Resolver arbitration. Used for intermediate outputs, draft analysis, and pipeline screening where speed matters.

SINGLE

Producer only. No QA review. Used only for exploratory analysis, rapid triage, and non-deliverable work product.

Quality Gates

Every phase output passes through quality gates before downstream consumption. Gates verify: arithmetic correctness (recalculation from raw inputs), internal consistency (cross-reference between sections), source attribution (every claim traced to data), and constraint compliance (no policy violations).

Audit Trail

The skill chain audit system produces a mandatory execution log for every run. Logs capture: skill name, execution timestamp, input parameters, output summary, QA findings, and resolution status. Logs are stored in /skill_chain_logs/ and are immutable once written.

Deployment

Get running in five steps.

Technical Requirements

  • Claude Code CLI (Anthropic) with active API key
  • Node.js 18+ for MCP data services
  • Python 3.10+ for report generation engine
  • API keys: Google Maps, Census, FRED, BLS, HUD, Walk Score
  • Windows, macOS, or Linux

Installation Steps

STEP 1

Clone the repository

Clone from GitHub and install dependencies. The repo includes all skills, engines, templates, and configuration files.

STEP 2

Configure API keys

Set up your .env file with API keys for data services: Google Maps, Census, FRED, BLS, HUD, and Walk Score.

STEP 3

Initialize your credit box

Edit credit_box_config.json to reflect your firm's constraint thresholds, sizing parameters, and credit philosophy.

STEP 4

Register MCP server

Register the valeri-data MCP server in your Claude configuration for integrated data service access during analysis.

STEP 5

Run your first deal

Start with the Context Brief (Phase -1) and progress through the execution framework. The system guides you through each phase.

Credit Box Configuration

The credit box is the central configuration file that governs all constraint thresholds and sizing parameters. Here is a representative example:

credit_box_config.json JSON 
{
  "version": "3.0",
  "constraints": {
    "ltv_as_is":        { "max": 0.75, "type": "hard_ceiling" },
    "ltv_stabilized":   { "max": 0.75, "type": "hard_ceiling" },
    "dscr_stabilized":  { "min": 1.05, "type": "hard_floor"   },
    "dy_stabilized":    { "min": 0.07, "type": "hard_floor"   },
    "ltc":              { "max": 0.80, "type": "hard_ceiling" },
    "max_loan_amount":  100000000
  },
  "sizing": {
    "method": "constraint_waterfall",
    "binding_constraint": "minimum_of_all"
  }
}

Environment Considerations

pinchfin is designed for local execution. All data processing occurs on your machine. No deal data is transmitted to external servers beyond the API calls you configure. The system operates with full data sovereignty.

For production deployment at scale, consider: API rate limiting configuration, concurrent session management, storage allocation for deal records and generated documents, and backup procedures for the master repository.
